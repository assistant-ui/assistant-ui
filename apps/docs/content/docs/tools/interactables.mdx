An interactable is a tool UI with live, shared state: the user edits it through React, the model edits it through an auto-generated tool, and both sides always see the latest value. An editable artifact, a notepad, a task board, or a canvas; anything the user and the assistant work on together, rather than the assistant just emitting into the chat.

<InteractableSample />

## Overview

There are two ways to define an interactable, depending on who creates it:

- **App-scoped:** a component you mount yourself with `useInteractable`, anywhere in your app (a sidebar, a panel, wherever). The model automatically gets an `update_{name}` tool to read and edit it, and its state can persist across threads.
- **Thread-scoped:** an interactable tool UI the model summons into the conversation. You define it with `interactableTool` inside `defineToolkit`, and it renders inline when the model calls it. Its state rides the thread's history.

Either way, an interactable is more than an ordinary tool UI:

- **Shared, editable state**: the user (via React) and the model (via the auto-generated `update_{name}` tool) both write to the same state, and each sees the other's edits
- **Partial updates**: the model sends only the fields it wants to change; the rest are preserved
- **Versioned**: each user edit and model `update_*` is recorded as a version you can display and `restore()` ([Versions](#versions))
- **Persistent**: state outlives the tool call and the turn, and survives a reload
- **One tool per name**: every instance of a name shares a single `update_{name}` tool, addressed by `id`

Common use cases:

- Task boards that the AI can add items to
- Data dashboards that update based on conversation
- Forms that the AI pre-fills
- Canvas/editor components that the AI can manipulate
- Artifacts and notepads the AI drafts in the thread and the user edits in place

A component you mount yourself. Define it with `useInteractable` where you render it (a sidebar, a panel, wherever). The model automatically gets an `update_{name}` tool to read and edit it.

<Callout type="warn">
  Define the `stateSchema` and `initialState` **outside** the component (or

```tsx
import { useInteractable } from "@assistant-ui/react";
import { z } from "zod";

const taskBoardSchema = z.object({
  tasks: z.array(
    z.object({
      id: z.string(),
      title: z.string(),
      done: z.boolean(),
    }),
  ),
});

const taskBoardInitialState = { tasks: [] };

function TaskBoard() {
  // Like useState, but the AI can also read and update it. The state type is
  // inferred from the schema.
  const [state, { setState }] = useInteractable("taskBoard", {
    description: "A task board showing the user's tasks",
    stateSchema: taskBoardSchema,
    initialState: taskBoardInitialState,
  });

  return (
    <div>
      <h2>Tasks</h2>
      <ul>
        {state.tasks.map((task) => (
          <li key={task.id}>
            <label>
              <input
                type="checkbox"
                checked={task.done}
                onChange={() =>
                  setState((prev) => ({
                    tasks: prev.tasks.map((t) =>
                      t.id === task.id ? { ...t, done: !t.done } : t,
                    ),
                  }))
                }
              />
              {task.title}
            </label>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

Then place it anywhere in your layout:

```tsx
function App() {
  return (
    <MyRuntimeProvider>
      <div className="flex">
        <Thread className="flex-1" />
        <TaskBoard /> {/* Placed anywhere in your app */}
      </div>
    </MyRuntimeProvider>
  );
}
```

import { defineToolkit, interactableTool } from "@assistant-ui/react";
import { z } from "zod";

const notepadSchema = z.object({
  title: z.string(),
  content: z.string(),
});

const toolkit = defineToolkit({
  notepad: interactableTool({
    description: "A notepad with drafted text the user can read and edit.",
    stateSchema: notepadSchema,
    render: ({ state, setState, version, streaming }) => (
      <NotepadView
        value={state}
        onChange={setState}
        busy={streaming}
        readOnly={version ? !version.isLatest : false}
      />
    ),
  }),
});
```

```tsx
const aui = useAui({

Thread-scoped state rides the thread's history, so it survives reloads with nothing extra to persist. ### Surface state to the model in your route

The `update_*` tools work out of the box, so the AI can already _write_ to an interactable. For it to _read_ the current state, your route handler has one required step. Each user message carries a frozen state snapshot in its metadata, but the AI SDK's `convertToModelMessages` ignores metadata. Call `injectInteractableContext` first to turn that snapshot into model-visible text:

```ts title="app/api/chat/route.ts"
import { openai } from "@ai-sdk/openai";
import { convertToModelMessages, streamText } from "ai";
import { injectInteractableContext } from "@assistant-ui/react-ai-sdk";

export async function POST(req: Request) {
  const { messages } = await req.json();

  const result = streamText({
    model: openai("gpt-5.4"),
    messages: await convertToModelMessages(injectInteractableContext(messages)),
  });

  return result.toUIMessageStreamResponse();
}
```

Each snapshot defaults to `[Current state of "note" (id: "n1"): {...}]`. When only some fields changed since the model last saw the state, the snapshot is a **diff**: it carries just the changed fields (`partial: true`) and reads as `[State of "note" (id: "n1") changed — updated fields: {...}]`, so a small edit to a large state doesn't re-send the whole thing. Pass a second `format` argument to control how snapshots read to the model. Handle both full and partial entries, and keep the instance id visible; the model uses it to address the `update_*` tool.

`injectInteractableContext` is part of `@assistant-ui/react-ai-sdk` and works on AI SDK `UIMessage`s. On a different backend (LangGraph, a custom runtime), this step is the same idea with your own injection; see [Other Runtimes](#other-runtimes).

Now when the user says _"Add a task called 'Buy groceries'"_, the AI sees the board's current state and calls the `update_taskBoard` tool to update it. Thanks to partial updates, the AI only needs to send the fields it wants to change.

## Scope

Each interactable has a `scope` that decides where its state lives and how it is re-seeded after a reload. It defaults to `"app"`, which is what the Quick Start uses.

- **`"app"`** (default): state is owned by the [persistence adapter](#persistence) and shared across every thread. It's included in `exportState`/`save`, and on reload it re-seeds from the saved adapter blob. Use this for app-wide UI the assistant manipulates regardless of which conversation is open (a settings panel, a shared dashboard).
- **`"thread"`**: state rides the thread's own message history instead. The snapshot stamped onto each sent message _is_ the persisted copy, so thread-scoped interactables are skipped by the persistence adapter (`save` and `exportState` ignore them) and re-seed from the latest snapshot in the current thread on reload. Use this for state that belongs to one conversation (a per-thread canvas or draft).

```tsx
useInteractable("draft", {
  description: "The draft being edited in this conversation",
  stateSchema: draftSchema,
  initialState: draftInitialState,
  scope: "thread",
});
```

<Callout type="warn">
  For `scope: "thread"`, only **sent** state reaches history. A live edit the
  user makes but never sends survives in-session remounts (scrolling, list
  virtualization) but is **lost on a full page reload**: the interactable
  re-seeds from the model-known thread record (its creating tool call's args,
  sent snapshots, and the model's own `update_*` calls), or from `initialState`
  if none exists. If an edit must outlive a reload before the next message,
  send it or use `scope: "app"` with a persistence adapter.
</Callout>

## Interactables in Messages

An interactable doesn't have to live in your layout. A tool call can _be_ the interactable: the model drafts a document, the user edits it right there in the thread, and later model edits render where they happen. Render it inline like a notepad, or out-of-line like an artifact panel.

`interactableTool` defines the whole tool as one toolkit entry: the entry key is the interactable name, and the tool's arguments are the initial state. You write one render function; instance wiring, streaming previews, and rendering the model's `update_{name}` calls are handled:

```tsx
import { defineToolkit, interactableTool } from "@assistant-ui/react";
import { z } from "zod";

const notepadSchema = z.object({
  title: z.string(),
  content: z.string(),
});

const toolkit = defineToolkit({
  notepad: interactableTool({
    description: "A notepad with drafted text that the user can read and edit.",
    stateSchema: notepadSchema,
    render: ({ state, setState, version, streaming }) => (
      <NotepadView
        value={state}
        onChange={setState}
        busy={streaming}
        readOnly={version ? !version.isLatest : false}
      />
    ),
  }),
});
```

That's the whole setup. When the model later calls the auto-generated `update_notepad`, the same `render` appears at the message that made the edit. In-message interactables are thread-scoped: state rides the thread's history (see [Scope](#scope)).

### Versions

A thread is an append-only log, so one instance accumulates **versions**: the creating call, each user edit, and each `update_*` call. Your `render` runs at every message that touched the instance and receives two things:

- `state` / `setState`: the **live** instance. There is exactly one, shared by every message.
- `version`: **this message's** version, `{ state, isLatest, restore }`. `version.state` is the interactable as it was at this point in the conversation; `restore()` sets the live state back to it.

What older messages do with that is your choice, and both common policies are one-liners:

```tsx
// Frozen history:
// old messages show their version read-only; only the newest is live
render: ({ state, setState, version }) =>
  version && !version.isLatest ? (
    <Notepad value={version.state} readOnly />
  ) : (
    <Notepad value={state} onChange={setState} />
  );
```

```tsx
// Editable history:
// every message edits the live note; revert returns to this message's version
render: ({ state, setState, version }) => (
  <Notepad value={state} onChange={setState} onRevert={version?.restore} />
);
```

Versions are computed from the thread record itself (the creating call's args, the state snapshots stamped on sent user messages, and the model's `update_*` calls), so history survives reloads with nothing extra to persist.

The `version` prop above is per-message, for what each message renders. To list every version at once, for a history dropdown that works for both scopes, see [Version History](#version-history).

### Streaming

While a tool call's arguments are still streaming, `render` is called with `streaming: true`: at the creating call `state` is the partially generated draft (fields may be missing), and at an `update_{name}` call it is the live state updating as the edit streams in. Render a preview; edits made during a create stream are dropped.

### Artifacts

Artifacts are usually read-only: the model produces a document, and the user can only view it. An interactable artifact goes a step further, since the user can edit it too and the model sees those edits. It is the same pattern with the rendering split in two: the inline tool UI renders a **button**, and the full interactable lives in a panel (sidebar, dialog) in your layout.

```
inline messages                         artifact panel (layout)
───────────────────────────            ─────────────────────────────
render: a button per version           useInteractable(name, {id,
  "📝 Meeting notes" → onOpen(id)        scope: "thread"}) → live,
                                         editable, full height
```

```tsx
const toolkit = defineToolkit({
  document: interactableTool({
    description: "A document the user can open and edit.",
    stateSchema: documentSchema,
    render: ({ state, version, id }) => (
      <ArtifactButton
        title={(version?.state ?? state).title}
        onClick={() => openArtifact(id)}
      />
    ),
  }),
});

function ArtifactPanel({ id }: { id: string }) {
  const [state, { setState }] = useInteractable("document", {
    id,
    description: "A document the user can open and edit.",
    stateSchema: documentSchema,
    initialState: emptyDocument,
    scope: "thread",
  });
  const versions = useInteractableVersions<Document>(id, "document");

  return (
    <aside>
      <VersionDropdown>
        {versions.map((v, i) => (
          <DropdownItem key={i} onSelect={v.restore}>
            v{i + 1}: {v.origin === "user-edit" ? "you" : "assistant"}
          </DropdownItem>
        ))}
      </VersionDropdown>
      <Editor value={state} onChange={setState} />
    </aside>
  );
}
```

Outside message parts the hook always returns the live state (`version` is `undefined`), so the panel is plainly editable. Both the panel and the inline tool UIs can register the same `id`; registration is shared.

<Callout type="warn">
  Keep the artifact panel mounted (hidden is fine) whenever its instance should
  stay available. If every component registering the instance unmounts (panel
  closed and inline buttons virtualized out of the thread), the instance and
  its `update_{name}` tool unregister, and the tool list churns.
</Callout>

### How it's wired

`interactableTool` is built on two low-level pieces, available as an escape hatch:

- **The `id: toolCallId` convention.** The creating tool call's `toolCallId` doubles as the instance id, which also tells the snapshot channel the model authored the initial state (the call's args), so nothing is re-stamped on the next send and state restores from thread history after a reload.
- **`updateRender` on `useInteractable`.** The `update_{name}` tool is auto-generated, so this config option installs its tool UI (once per name). Inside an `update_{name}` part the hook infers the instance `id` from the call, so the same component serves both call types.

```tsx
const NotepadToolUI: ToolCallMessagePartComponent<NotepadArgs> = ({
  toolCallId,
  args,
  result,
}) => {
  if (!result) return <DraftPreview args={args} />;
  return <Notepad id={toolCallId} initial={args} />;
};

function Notepad({ id, initial }: { id?: string; initial: NotepadArgs }) {
  const [state, { setState, version }] = useInteractable("notepad", {
    id,
    description: "A notepad the user can read and edit.",
    stateSchema: notepadSchema,
    initialState: initial,
    scope: "thread",
    updateRender: NotepadUpdateToolUI, // gate on `result` before mounting this
  });
  // ...
}
```

## Version History

Every interactable accumulates **versions** as it changes: each user edit, each model `update_*` call, and (for thread-scoped) the creating call. `useInteractableVersions(id, name)` returns them oldest-first, each with the full `state` as of that version and a `restore()` that sets the live instance back to it. You can use it to create a versions dropdown:

```tsx
function VersionDropdown({ id, name }: { id: string; name: string }) {
  const versions = useInteractableVersions(id, name);
  if (versions.length < 2) return null;

  return (
    <select onChange={(e) => versions[+e.target.value]!.restore()}>
      {versions.map((v, i) => (
        <option key={i} value={i}>
          v{i + 1}: {v.origin === "user-edit" ? "you" : "assistant"}
        </option>
      ))}
    </select>
  );
}
```

**Thread-scoped:**

```tsx
const Notepad = ({ id, state, setState }: InteractableToolRenderProps<NotepadArgs>) => (
  <div>
    <VersionDropdown id={id} name="notepad" />
    {/* ...editor... */}
  </div>
);
```

**App-scoped:**

```tsx
const [state, { id }] = useInteractable("taskBoard", config);
return <VersionDropdown id={id} name="taskBoard" />;
```

Versions are read from the current thread's messages, so an app-scoped item's history covers the current conversation, not its full cross-thread lifetime.

## Persistence

By default, `scope: "app"` interactable state is in-memory and lost on page refresh. You can add persistence by passing an adapter to `Interactables`:

```tsx
import { useAui, Interactables } from "@assistant-ui/react";

// Module-level (or memoized) so the adapter identity is stable across renders.
const persistenceAdapter = {
  load: () => {
    const saved = localStorage.getItem("interactables");
    return saved ? JSON.parse(saved) : undefined;
  },
  save: (state) => {
    localStorage.setItem("interactables", JSON.stringify(state));
  },
};

function MyRuntimeProvider({ children }) {
  const aui = useAui({
    interactables: Interactables({ persistence: persistenceAdapter }),
  });

  return /* ... */;
}
```

`load` is called when the adapter is attached and may be async. Loaded state seeds interactables as they register; a local edit made while a slow `load` is still in flight wins over the loaded value. Thread-scoped interactables are not touched by the adapter; they persist via thread history (see [Scope](#scope)).

For dynamic setups (an adapter that depends on auth), call `aui.interactables().setPersistenceAdapter(adapter)` imperatively instead.

### Sync Status

When a persistence adapter is set, interactable hooks expose sync metadata:

```tsx
const [state, { setState, isPending, error, flush }] = useInteractableState<TState>(id);

// isPending: true while a save is in-flight
// error: the error from the last failed save, if any
// flush(): force an immediate save (useful before navigation)
```

State changes are automatically debounced (500ms) before saving. When the owning component unmounts, any pending save is flushed immediately.

### Export / Import

For custom persistence strategies, use `exportState` and `importState` directly:

```tsx
const snapshot = aui.interactables().exportState();
// => { "note-1": { name: "note", state: { title: "Hello" } }, ... }

aui.interactables().importState(snapshot);
// Imported state is picked up when components next register
```

### Schema Evolution

<Callout type="warn">
  If you change a Zod schema after state has been persisted, the loaded snapshot may silently mis-match the new shape. The adapter does a shallow merge, so extra fields are preserved and missing fields keep their initial values, but type mismatches are not caught at runtime. To avoid silent corruption, version your schema key (e.g. `"taskBoard_v2"`) or namespace it by schema hash whenever you make breaking changes. Alternatively, add a migration step in your `load` or `importState` call.
</Callout>

## Streaming Updates

While the AI is generating tool arguments, the interactable's state updates in real time as fields stream in. You can combine the partial state with the thread's running status to show a skeleton UI while the AI is mid-stream:

```tsx
import { useAuiState } from "@assistant-ui/react";

function TaskBoard() {
  const [state] = useInteractable("taskBoard", config);
  const isRunning = useAuiState((s) => s.thread.isRunning);

  const isLoading = isRunning && state.tasks.length === 0;

  return (
    <div>
      <h2>Tasks</h2>
      {isLoading ? (
        <div className="animate-pulse h-8 rounded bg-muted" />
      ) : (
        <ul>
          {state.tasks.map((task) => (
            <li key={task.id}>{task.title}</li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

The state object updates progressively as each field streams in, so partial renders work without any extra wiring. Use the thread's `isRunning` to distinguish "still streaming, no fields yet" from "model returned an empty result".

## Multiple Instances

You can render multiple interactables with the same `name` but different `id`s. They all share a single `update_{name}` tool. The model addresses an instance with the tool's `id` parameter, which it reads from the state snapshots in the conversation:

```tsx
import { useInteractable } from "@assistant-ui/react";
import { z } from "zod";

const noteSchema = z.object({
  title: z.string(),
  content: z.string(),
  color: z.enum(["yellow", "blue", "green", "pink"]),
});

const noteInitialState = { title: "New Note", content: "", color: "yellow" as const };

function NoteCard({ noteId }: { noteId: string }) {
  const [state] = useInteractable("note", {
    id: noteId,
    description: "A sticky note",
    stateSchema: noteSchema,
    initialState: noteInitialState,
  });

  return <div>{state.title}</div>;
}

function App() {
  return (
    <>
      <NoteCard noteId="note-1" />
      <NoteCard noteId="note-2" />
      {/* one update_note tool: update_note({ id: "note-2", color: "blue" }) */}
    </>
  );
}
```

The tool's name, schema, and description never change as instances mount and unmount, so the model's tool list (and provider prompt caches) stay stable. While exactly one instance exists, the model may omit `id`. A call with an unknown `id` returns an error listing the valid ids, so the model can recover.

<Callout type="info">
  A top-level `id` field in your `stateSchema` is reserved: the update tool
  uses it for instance addressing, so the model cannot write a state field
  named `id`. Nest it or name it differently (e.g. `noteId`).
</Callout>

## Partial Updates

Auto-generated tools use a partial schema in which all fields are optional. The AI only sends the fields it wants to change; omitted fields keep their current values.

```tsx
// If the state is { title: "My Note", content: "Hello", color: "yellow" }
// The AI can call: update_note({ color: "blue" })
// Result: { title: "My Note", content: "Hello", color: "blue" }
```

This is especially useful for large state objects where regenerating the entire state would be expensive and error-prone.

<Callout type="info">
  Merge is shallow (one level deep). If the AI sends a nested object, it replaces
  that entire field rather than deep-merging into it.
</Callout>

## Selection

Selection is not a special concept; it's just state. Earlier versions exposed a top-level `selected` prop and `setSelected` method, but selection is now an ordinary field in your `stateSchema` (see [Migrating from the Previous API](#migrating-from-the-previous-api)). Add a `selected` field to the schema and let the parent own which `id` is selected. Every card keeps its own `selected` flag in sync from that single source of truth:

```tsx
const noteSchema = z.object({
  title: z.string(),
  content: z.string(),
  color: z.enum(["yellow", "blue", "green", "pink"]),
  selected: z.boolean(),
});

const noteInitialState = {
  title: "New Note",
  content: "",
  color: "yellow" as const,
  selected: false,
};

function NoteCard({
  noteId,
  selectedId,
  onSelect,
}: {
  noteId: string;
  selectedId: string | null;
  onSelect: (id: string) => void;
}) {
  const [state, { setState }] = useInteractable("note", {
    id: noteId,
    description: "A sticky note",
    stateSchema: noteSchema,
    initialState: noteInitialState,
  });

  const isSelected = selectedId === noteId;

  // Each card self-syncs its `selected` field from the parent's selectedId, so
  // exclusive selection stays a single source of truth with no cross-card writes.
  useEffect(() => {
    if (state.selected !== isSelected) {
      setState((prev) => ({ ...prev, selected: isSelected }));
    }
  }, [isSelected, state.selected, setState]);

  return <div onClick={() => onSelect(noteId)}>{state.title}</div>;
}
```

The parent passes `selectedId` and `onSelect={setSelectedId}` to each card. Because `selected` lives in the schema, the AI sees which note is focused: _"Change the color to blue"_ targets the right note, and _"select the meeting-prep note"_ works too.

## How It Works

1. **Registration**: the interactable is registered in the `interactables` scope with its name, description, schema, and initial state.
2. **Tool generation**: an `update_taskBoard` frontend tool is created with a partial schema (all fields optional) plus a required `id` parameter. One tool exists per name no matter how many instances mount, so the tool list stays stable and provider prompt caches stay warm.
3. **State snapshot**: the interactable's current state is stamped onto the outgoing user message as a frozen snapshot in `metadata.custom` (no system-prompt mutation). It is stamped only when the model doesn't already know the state: a user edit triggers a fresh snapshot, while the model's own `update_*` calls (and the args of the tool call that created an [in-message instance](#interactables-in-messages)) don't, since the model authored those. When the change fits a shallow merge, the snapshot carries only the changed fields (`partial: true`) instead of cloning the full state. Your route turns snapshots into model-visible text with `injectInteractableContext`; see [Surface state to the model](#surface-state-to-the-model-in-your-route).
4. **Streaming updates**: as the AI generates the tool arguments, the interactable's state updates progressively rather than waiting for complete arguments. This gives users immediate visual feedback.
5. **Partial merge**: only the fields the AI sends are updated; the rest are preserved.
6. **Bidirectional updates**: when the AI calls the tool, the state updates and React re-renders. When the user updates state via `setState`, the next user message carries a fresh snapshot so the AI sees the latest state.

## Unmount Behavior

When a component that called `useInteractable` unmounts, the interactable is unregistered, but its state is preserved in the `Interactables` scope. When the component mounts again with the same name and id, the scope restores the preserved state rather than resetting to `initialState`. This means transient unmounts (such as React Strict Mode double-mounts or tab switches) do not lose state.

An instance registered from several places (its creating tool call, `update_*` calls, an artifact panel) stays registered until the last one unmounts, so scrolling one out of a virtualized thread doesn't tear the instance down while another is visible.

## Other Runtimes

`injectInteractableContext` is AI SDK-specific, but the snapshot channel itself is not: every user message carries its pending snapshots at `metadata.custom.interactables`. For other backends (LangGraph, a custom runtime), build the equivalent injection with the two helpers exported from `@assistant-ui/react`:

```ts
import {
  getInteractableSnapshots, // (message) => snapshot entries | undefined
  formatInteractableSnapshot, // (entry) => the default model-facing wording
} from "@assistant-ui/react";

for (const message of messages) {
  if (message.role !== "user") continue;
  const items = getInteractableSnapshots(message);
  if (!items?.length) continue;
  const text = items.map(formatInteractableSnapshot).join("\n");
  // prepend `text` to the message content in whatever shape your backend expects
}
```

Three rules to preserve: only transform a copy of the messages on the way to the model (snapshots must stay raw in stored history, since the send-time gate diffs against them), keep each instance's `id` visible in the wording so the model can address the `update_*` tool, and handle `partial: true` entries, whose `state` carries only the fields that changed (`formatInteractableSnapshot` does).

### `useInteractableVersions`

```tsx
const versions = useInteractableVersions<TState>(id, name);
// → [{ state, origin: "create" | "update" | "user-edit", toolCallId?, restore }, ...]
```

Every version of an interactable recorded in the current thread, oldest first. Each entry carries the full `state` and a `restore()` that sets the live instance back to it. Works for both scopes; see [Version History](#version-history) for usage. The non-React equivalent for backends is `getInteractableVersions(messages, id, name)`, exported from `@assistant-ui/react`.

### `Interactables`

The scope resource that manages all interactables. Register it via `useAui`, optionally with a [persistence adapter](#persistence):

```tsx
const aui = useAui({
  interactables: Interactables({ persistence: myAdapter }),
});
```

## Migrating from the Previous API

If you used an earlier version of the interactables API:

- `useAssistantInteractable` and `useInteractableState` have been merged into a single [`useInteractable`](#useinteractable) hook that registers and returns state. `useInteractableState` remains for secondary readers.
- Per-instance tools (`update_note_note-1`) are gone. Each name has one stable `update_{name}` tool with a required `id` parameter.
- The top-level `selected` prop and `setSelected` method have been removed. Represent selection as ordinary state; see [Selection](#selection).
